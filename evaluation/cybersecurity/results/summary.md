## 发现的问题

### 1. 第一版评测集只能作为冒烟基线

四种检索模式都达到满分，说明当前语料主题区分度较高、规模太小，尚不能证明混合检索或
Rerank 优于单路召回。下一版需要加入更多同主题文档、跨文档重复术语、长文档、表格、扫描 PDF、
旧版本政策和仅有细微差异的 CVE 公告。

### 2. Rerank 当前没有带来质量收益

在本轮数据上，Rerank 没有提高 Hit、MRR 或 nDCG，却使平均延迟约增加一倍。初次运行还出现过
一次 Rerank API `ConnectError`：请求约 19.4 秒后降级为融合结果，相关证据仍排第 1。复测未复现，
说明降级逻辑有效，但外部 API 的连接超时会明显拉高尾延迟。

建议后续扩大难例后再判断是否保留 Rerank，并分别评估更短连接超时、有限重试和熔断策略。

### 3. 不可回答问题仍固定返回结果

开发集 3 条、测试集 1 条不可回答问题均返回 10 条结果。检索器的职责是排序候选，它目前没有
可校准的“无答案”阈值；RRF 分数也不适合直接跨查询比较。因此本轮没有把不可回答问题混入
Hit/MRR 计算。

后续需要在回答层增加证据充分性判断和拒答策略，再评估拒答准确率、无证据断言率和引用正确率。

### 4. Gold 需要持续复核

首轮开发集曾把 `CYB-033` 的 ATT&CK 攻击链概览页漏标为相关证据。人工复核后已补充 `ATT-004`，
并通过 `--reuse-pdf` 只重建标注，没有改变已上传 PDF。后续扩充数据集时应进行双人标注或至少
使用候选池复核，避免把合理召回误判为失败。

## 当前结论

真实 Qdrant + OpenSearch + 百炼 Embedding/Rerank 的端到端链路已跑通，现有小型语料上的基础召回
没有问题。下一阶段的重点不是继续调参数，而是扩充高混淆度语料和问题，使评测能够真正区分
BM25、向量检索、RRF 和 Rerank 的贡献。
